# CodeAgent Build Architecture

CodeAgent is being refactored from an inherited terminal-first codebase into an Electron desktop app. The active desktop build is split into Electron main/preload/services, a browser renderer, and package artifacts generated by `electron-builder`.

---

## Desktop Build Pipeline

```text
TypeScript source
      |
      | npm run build:electron:main
      v
dist-electron/
      |
      | tsc -p tsconfig.renderer.json --noEmit
      v
renderer typecheck
      |
      | node build-renderer.mjs
      v
dist-renderer/
      |
      | electron-builder
      v
dist-build/
```

The default desktop validation command is:

```bash
npm run build:electron
```

The local package validation command is:

```bash
npm run pack
```

## Runtime Processes

### Electron Main Process

Source:

```text
electron/main.ts
electron/bridge.ts
electron/services-bridge.ts
electron/services/*.ts
```

Responsibilities:

- App lifecycle and single-instance behavior.
- BrowserWindow creation and window state persistence.
- Native menus, dialogs, app identity, and updater integration.
- IPC registration for desktop services.
- Main-process access to filesystem, auth/keychain, MCP discovery, command execution, automation, and local history.

### Preload Process

Source:

```text
electron/preload.ts
electron/types.ts
```

Responsibilities:

- Expose a narrow, typed `window.api` surface to the renderer.
- Keep renderer sandboxing enabled.
- Prevent direct Node.js access from renderer code.

### Renderer

Source:

```text
src/renderer/App.tsx
src/renderer/App.module.css
src/renderer/ipc-client.ts
src/renderer/index.html
```

Responsibilities:

- Desktop workbench UI for chat, projects, tools, automation, history, and settings.
- Typed IPC client calls into Electron service bridges.
- Tool activity, permission review, command runner, file browser, MCP registry, local history, and automation views.

## Service Bridge Layer

Electron service bridges wrap local capabilities behind IPC-safe interfaces:

```text
tool-service-bridge.ts
api-service-bridge.ts
filesystem-service-bridge.ts
auth-service-bridge.ts
app-state-service-bridge.ts
mcp-service-bridge.ts
command-service-bridge.ts
web-service-bridge.ts
finance-service-bridge.ts
automation-service-bridge.ts
local-history-service-bridge.ts
```

The renderer imports `src/renderer/ipc-client.ts`, not Electron or Node APIs directly.

## Storage Layers

Workspace-shareable state:

```text
.code-agent/project.json
.code-agent/skill-policies.json
.code-agent/tasks/
.code-agent/teams/
.code-agent/runs/
```

Local-only workspace state:

```text
.code-agent/local/
```

User profile state:

```text
Electron userData
```

Secrets are owned by the OS keychain when available. They must not be copied into project bundles, history exports, or sync payloads.

See `docs/storage-ownership.md` for the full ownership model.

## Packaging Architecture

Packaging uses `electron-builder` from the `build` field in `package.json`.

Key package metadata:

- App id: `com.albertforweb.codeagent`
- Product name: `CodeAgent`
- Build resources: `electron/resources`
- Output directory: `dist-build`
- Publish provider: GitHub Releases

Brand assets:

```text
electron/resources/codeagent-logo.svg
electron/resources/icon.icns
electron/resources/icon.ico
electron/resources/icon.png
```

The asset generator is:

```bash
npm run generate:brand-assets
```

## Legacy CLI Build Note

The repository still contains a large inherited CLI source tree with Bun-specific APIs and internal-source assumptions. That tree is valuable source material, but the broad legacy TypeScript build is not the acceptance check for desktop work.

Use these commands for current desktop validation:

```bash
npm run build:electron
npm run pack
```
